# Co-op Coverage Audit - 2026-07-07

A full sweep of the decompiled game (~437 files) against the mod's sync coverage,
run before the 1.0.29 release. 22 agents: 2 inventory, 10 domain sweeps, 10 adversarial
verifiers. Every confirmed gap below survived an independent refutation attempt.

**Stats:** 18 gap claims -> 10 verified -> 10 confirmed real, 0 refuted; 134 systems confirmed covered.

## Confirmed gaps (fixed in 1.0.29 unless noted)

### [CRITICAL] Auto pack opener (InteractableAutoPackOpener) — join-time save load re-forwards stored packs (cards-packs)

On the guest's join-time world load, the game's own save-restore path InteractableAutoPackOpener.LoadData (decompiled/InteractableAutoPackOpener.cs:370-411) calls AddItem(item, addToFront:true, isPlayer:false) once per pack that was stored in the host's opener at save time. The mod's Harmony prefix ContainerSync.PackOpenerAddItemPrefix (src/CardShopCoop/Sync/ContainerSync.cs:863-885) fires for every one of those AddItem calls because its ONLY guard is `CoopCore.Role != CoopRole.Client || ApplyingRemote` — during ApplyAndLoad the role is already Client (set at CoopCore.cs:389/1154 before the load at CoopCore.cs:2474) and ApplyingRemote is false (this is the game's loader, not a ContainerSync apply). Each firing sends OpPackInsert to the host via SendOp (unconditional Send to connId 1, CoopCore.cs:308/1657 — no in-game/preload gate). The host's HostApplyOp OpPackInsert branch (ContainerSync.cs:361-374) spawns a brand-new pack Item into its own opener UNCONDITIONALLY (the comment even says 'apply unconditionally... dropping it would eat the pack'). The host already had those packs from its own live opener state, so every stored pack is DUPLICATED on the host once per join. ClientPreloadHold only gates the periodic ClientTick snapshot detectors (CoopCore.cs:285/803-804), not this Harmony prefix; MsgType.ContainerOp dispatch (CoopCore.cs:3123-3127) has no 'guest still loading' guard.

**Fix:** Gate PackOpenerAddItemPrefix on `!CoopCore.ClientReloading`, exactly matching the existing CardBoxSync.cs:172 / FurnBoxSync.cs:222 pattern. Change the guard at ContainerSync.cs:865 from `if (CoopCore.Role != CoopRole.Client || ApplyingRemote) return true;` to `if (CoopCore.Role != CoopRole.Client || ApplyingRemote || CoopCore.ClientReloading) return true;` so AddItem calls originating from the game's save-restore during a join are treated as replay, not player inserts. Apply the same guard by symmetry to CleanserAddItemPrefix (ContainerSync.cs:954), whose AddItem is also invoked from the analogous cleanser LoadData restore path. Alternatively (broader fix) wrap the guest's world load so all interactable LoadData mutations are recognized as replay, but the per-prefix ClientReloading guard is the minimal change consistent with the codebase's existing approach.

### [CRITICAL] Sell / relocate an EXISTING placed shelf or counter via box-up (guest) (layout)

A guest can box up an existing HOST-placed furniture object (StartMoveObject -> BoxUpObject creates a new InteractablePackagingBox_Shelf in the guest's RestockManager.m_ShelfPackagingBoxList) and then SELL it (InteractionPlayerController.ConfirmSellFurniture: fires CEventPlayer_AddCoin(price/2) then box.OnDestroyed). The AddCoin is forwarded to the shared wallet via DayEndBlockPrefix->EconContrib, crediting the shared bank. But the box the guest created is guest-only; the host never boxed that object. FurnBoxSync.HostApplyRemoved -> FindBox looks in the HOST's m_ShelfPackagingBoxList and finds no match (returns null) -> removal ignored. The host's shelf survives and its next PopState re-spawns it on the guest. Net: shared wallet gains the sell price, the furniture is NOT removed.

**Fix:** Patch InteractionPlayerController.ConfirmSellFurniture on the CLIENT: block-before-charge (prefix returns false locally so no local AddCoin and no local box.OnDestroyed) and instead forward a 'sell existing furniture' op identifying the boxed object by (kind,index)/identity + type, which the host validates against a REAL placed object it owns, then the host performs the single AddCoin AND the object's OnDestroyed so PopulationSync mirrors the removal to everyone. Additionally, forward the initial BoxUpObject(holdBox:true) of an existing placed object (patch InteractableObject.BoxUpObject on the client, distinguishing a placed object being boxed from a delivery unpack) so relocate-via-box mirrors symmetrically and the host's own copy is boxed/held in step, closing the same class of gap for relocation as well as selling.

### [MODERATE] Restock / furniture / license purchases (guest-forwarded) (economy)

For a GUEST purchase, the coin charge and the product are two DECOUPLED forwarded messages: the guest fires CEventPlayer_ReduceCoin (forwarded as EconContrib kind 2, which the host affordability-guards and can DECLINE) and, separately, the product order/furniture/license (ForwardOrder / ForwardFurniture / ForwardLicense, which the host applies UNCONDITIONALLY with no coin check). When the host declines the charge because the shared wallet is short, it still delivers the product for free. This is structural, not a rare race: the guest's local wallet mirror does NOT decrement on its own forwarded ReduceCoin (DayEndBlockPrefix returns false without applying locally, GamePatches.cs:398-402), so it only refreshes on the ~0.5s CoinSet echo. A guest making several rapid purchases all pass their own local affordability check against the same stale balance; the host's _pendingReduceThisFrame guard (CoopCore.cs:3419-3427) then correctly DECLINES the ones that overrun the real wallet — but the corresponding forwarded products (OrderRequest CoopCore.cs:2912, FurnitureOrder CoopCore.cs:2853, LicenseUnlock CoopCore.cs:2988) are applied regardless. Restock and furniture ALSO fire a forwarded AddShopExp, so the guest even gains XP for the unpaid item.

**Fix:** Adopt the grading/hire/unlock coupling for the three paths. Preferred: block the guest's local purchase BEFORE its ReduceCoin (prefix that returns false like StaffSync/ShopStateSync do) and forward a single op whose host handler re-checks affordability from CPlayerData.m_CoinAmountDouble and, only if it passes, atomically issues the host-side ReduceCoin AND delivers the product+XP. Minimal alternative without restructuring the wire protocol: in the EconContrib kind-2 decline branch (CoopCore.cs:3420-3424), record the declined amount per ConnId for this dispatch frame, and have the OrderRequest/FurnitureOrder/LicenseUnlock handlers, when they see a matching just-declined charge from the same peer this frame, refuse+toast instead of delivering (or, since the handlers already know the authoritative price — lineCost/fp.price/license price — refund the product's cost the way the catalog-miss branch already does).

### [MODERATE] Decoration objects (posters / statues / arrow-signs / vertical plants) - guest placement (layout)

A guest placing a deco OBJECT (EDecoObject, category-3 deco: 72 posters, 35 statues, 13 arrow signs, 8 vertical/wall plants) has no forward path. PlaceDecoUIScreen.StartPlaceDecoItem spawns the object into the guest's m_DecoObjectList (kind 5) and does AddDecoItemToInventory(-1) purely locally. PopulationSync is host-authoritative (host broadcasts its kind-5 roster, client reconciles TO it), so the guest's new deco is an 'extra beyond host roster' and gets destroyed by the client's next PopState reconcile. Meanwhile the guest already consumed its inventory count. Worse: ObjMoveSync.Tick runs on the client and will send an ObjMoveRequest keyed (kind5<<24|index) for the guest's settled deco; on the host ObjMoveSync.Resolve/ApplyRemote does an index lookup with NO type check (Resolve at ObjMoveSync.cs:189-196) and SetPositionAndRotation, so it teleports whatever DIFFERENT host deco happens to sit at that kind-5 index.

**Fix:** Minimal, matching the mod's existing block idiom (RenamerBlockPrefix / tournament blocks): add a client prefix on `PlaceDecoUIScreen.StartPlaceDecoItem` (and optionally `ShopBuyDecoUIScreen.OnPressBuyShopDecoItem`) that, when Role==Client, shows a register/toast line ('deco placement is host-only') and returns false BEFORE `AddDecoItemToInventory(-1)` runs, so the guest neither loses inventory nor spawns an orphan kind-5 object. Better (parity with furniture): forward StartPlaceDecoItem to the host (spawn EDecoObject on host via SpawnDecoObject, let PopulationSync mirror it back) and mirror m_DecoItemInventoryList deltas + include it in the join save. Independently, harden ObjMoveSync.Resolve/ApplyRemote with a per-kind type guard so a kind-5 request whose resolved object's m_DecoObjectType doesn't match the expected type is dropped (defense-in-depth against the stray-move teleport).

### [MODERATE] Smelly customers (handheld deodorant spray) (customers)

When the GUEST picks up a deodorant spray and sprays it at a smelly customer, nothing happens to the host's real smelly customer. InteractionPlayerController's hold-spray path (decompiled/InteractionPlayerController.cs:1628-1631) calls CustomerManager.GetCustomerList()[i].DeodorantSprayCheck(...) against the LOCAL customer list. On the guest that list is nothing but the suppressed/deactivated puppet customers (NpcSweepTick disables them, ClientBlockPrefix stops their sim) - Customer.DeodorantSprayCheck (decompiled/Customer.cs:532-558) reduces m_SmellyMeter and clears m_IsSmelly on that local puppet object, but the host's authoritative smelly customer (the one whose stink cloud the guest actually sees, mirrored via NpcSync's Smelly flag) is a DIFFERENT object on the host and is never touched. There is no MsgType or forward for a handheld spray hit.

**Fix:** Mirror the vanilla station model for the handheld. Add a client-side Harmony postfix (or transpiler guard) on InteractionPlayerController.RaycastHoldSprayState: on the guest, suppress the local GetCustomerList() DeodorantSprayCheck loop (it only hits inert puppets) and instead, when the spray VFX fires with content, forward a lightweight op to the host carrying the spray world position + range (2.5) + potency (1). Add a MsgType (e.g. HandheldSprayHit) and a dispatch case in CoopCore that, host-side, runs DeodorantSprayCheck against the host's REAL customer list at that position (the same call InteractableAutoCleanser.Spray uses); the existing NpcSync Smelly flag + ReportSync.smellyCustomerCleaned mirror then echoes the cleaned result back to the guest automatically. If handheld co-op spraying is judged low-value, the cheaper mitigation is to gate it on the client: block entering HoldSprayState with a toast ("the host or the auto-cleanser clears smelly customers") so the guest never wastes deodorant content on a guaranteed no-op.

### [MODERATE] Worker AI vs guest-carried delivery boxes (BoxSync interplay) (workers)

On the host, a box the GUEST is carrying is marked _remoteCarried and its host-side GameObject is SetActive(false), but the host never sets m_IsBeingHold or m_PreventWorkerTakeBox on that box. Worker.DetermineShopAction (RestockShelf/RefillCleanser/RefillCardOpener) builds its candidate list via RestockManager.GetItemPackagingBoxListWithItem, which filters on IsValidObject() && GetItemCount()>0. InteractableObject.IsValidObject() returns !m_IsBeingHold (decompiled/InteractableObject.cs:834-841) and does NOT check gameObject.activeSelf, and CanWorkerTakeBox() returns !m_PreventWorkerTakeBox. So a host worker still treats the guest-held box as a valid, takeable restock source.

**Fix:** On the host, exclude _remoteCarried boxes from worker consideration for the duration of the guest carry. When a box enters _remoteCarried (BoxSync.cs HostApplyRequest ~541 and the HostTick re-assert ~499), set m_PreventWorkerTakeBox=true (and/or m_IsBeingHold=true) on the host box via reflection so CanWorkerTakeBox()/IsValidObject() reject it; clear the flag when the guest releases it (_remoteCarried.Remove path ~542 and on set-down). Setting m_PreventWorkerTakeBox is the cleaner choice since m_IsBeingHold interacts with other host code paths. This mirrors the existing IsBeingHeld guard but for the guest-carry direction.

### [MODERATE] Save/persistence - manual Load Game from pause menu (guest) (time-save)

The mod blocks WRITES via SaveGuardPrefix on CGameManager.SaveGameData, but does NOT block a guest from LOADING a different save mid-session. PauseScreen.OnPressLoadGame -> SaveLoadGameSlotSelectScreen.OpenScreen(isSaveState:false) -> OnPressLoadGame -> CGameManager.LoadMainLevelAsync("Start", slot) loads the guest's OWN save into the game 'Start' scene. This is a scene change to a non-Title scene, so the mod's OnSceneLoaded Shutdown branch (which only fires on scene.name=="Title") never runs, and ClientReloading is NOT set (that flag is set only by the mod's own join reload at CoopCore.cs:2472). The guest stays Role=Client with the TCP/Steam socket still open; the host is never told the guest left.

**Fix:** Broaden CoopCore.OnSceneLoaded (CoopCore.cs:754): if Role==Client && !ClientReloading && a fresh game-level ("Start") scene loads, treat it as the guest leaving and call Shutdown("left the session")/Disconnect — this catches manual Load Game plus any other non-mod scene reload (e.g. Back To Menu), and is safer than only patching the button. Apply the equivalent guard for the host so guests aren't stranded. (An additional client/GuestBorrowedWorld prefix on SaveLoadGameSlotSelectScreen.OpenScreen or OnPressLoadGame that blocks the load with a "Leave the co-op session first" toast is a good complementary UX guard, but the OnSceneLoaded broadening is the load-bearing fix.)

### [MODERATE] Loose boxes / warehouse restock (BoxSync type-sanity guard) (restock-storage)

BoxSync.HostApplyRequest rejects any guest box-edit whose reported item type differs from the host box's CURRENT compartment type (`if ((int)box.m_ItemCompartment.GetItemType() != e.Type) continue;`, BoxSync.cs:535). The guest can legitimately CHANGE an open box's item type via InteractablePackagingBox_Item.RemoveItemFromShelf (decompiled line 280-297): right-clicking a stocked display/warehouse shelf while holding an EMPTY OPEN box makes the box's compartment adopt the shelf's item type (ShelfCompartment.CheckItemType sets the type when m_ItemAmount<=0, line 242-247) and pulls one item off the shelf into the box. On the guest this is two mutations: the shelf compartment loses 1 item (WorldSync, kind 0/3 - syncs to host fine) and the box gains 1 item of the newly-adopted type (BoxSync - REJECTED by the guard, because the host's mirror of that box is still type None). The shelf decrement reaches the host but the box gain never does, so on the host-authoritative world the item is destroyed. Within ~6s the host's next BoxState broadcast (box still None/0) overwrites the guest's _locallyTouched copy and the item disappears on the guest too.

**Fix:** In BoxSync.HostApplyRequest (BoxSync.cs:535), relax the type guard to only reject when BOTH the host box and the requested entry are non-None and disagree: e.g. `int htype=(int)box.m_ItemCompartment.GetItemType(); if (htype != e.Type && htype != (int)EItemType.None && box.m_ItemCompartment.GetItemCount() > 0) continue;` — allowing an empty/None host box to adopt the guest's reported type. Then let ApplyToBox(box, e) run: for an empty->typed transition it must set the compartment's item type before spawning count (mirror FillBoxWithItem: SetCompartmentItemType + CalculatePositionList, as ApplyClosedCount already does for the pos-list case). This makes the reclaimed item land on the host box so conservation holds. Alternatively, patch RemoveItemFromShelf on the client to forward an explicit reclaim op the host replays authoritatively (heavier, but avoids widening the guard for genuinely-mangled requests).

### [MODERATE] Grading — re-submit of an already-graded card (re-grade) (grading-mail)

When the guest re-submits a card that is ALREADY graded (selected from the graded album), the host removes that graded card from m_GradedCardInventoryList TWICE: once via the GradedRemove mirror (RemoveGradedCardPostfix -> ForwardGradedRemoval -> host ApplyCardDelta -> RemoveGradedCard) and again via GradingSync.HostApplyOp's own degrade loop (CoopCore/Sync/GradingSync.cs lines 425-432, unconditional RemoveGradedCard for any cardGrade>0). RemoveGradedCard(ignoreGradedCardIndex:true) matches purely by identity (cardSaveIndex+grade+expansion+isDestiny), so if the host owns >=2 identical graded copies, the second (uninvolved) copy is also destroyed. The line-429 RemoveGradedCard also runs WITHOUT GamePatches.ApplyingRemoteCards set, so its own postfix re-broadcasts a GradedRemove to all guests, making the same double-removal happen on any guest that owns a duplicate too.

**Fix:** Delete the degrade loop at GradingSync.cs lines 425-432. The RemoveGradedCardPostfix mirror (the 1.0.22 mirror the code documents at 314-315) already removes the graded card on the host and relays it to all guests at SELECTION time, exactly like every other RemoveGradedCard, so the loop is redundant. Also update the stale comments at lines 26-28 and 421-424 which still claim the selection removal 'is a direct list edit no mirror carries'. If a belt-and-suspenders removal is preferred over deletion, gate the loop on HasGradedCardInAlbum AND wrap it in GamePatches.ApplyingRemoteCards=true (try/finally) so it neither removes a copy the mirror already took nor re-broadcasts a second GradedRemove — but deletion is the simplest correct fix.

### [MODERATE] Smelly customers / hand-held deodorant spray (catch-all)

The guest cannot clean smelly customers with the hand-held Deodorant spray item. InteractionPlayerController.RaycastHoldSprayState (decompiled line 1612-1631) sprays by iterating CustomerManager.GetCustomerList() and calling Customer.DeodorantSprayCheck on each. On the guest, real customer sim is suppressed (CustomerManager.Update ClientBlockPrefix, NpcSweepTick deactivates stray NPCs) and the visible customers are NpcSync puppets stored in a private _puppets dictionary with their Customer MonoBehaviour DestroyImmediate'd during the strip (NpcSync.Spawn, src/CardShopCoop/Sync/NpcSync.cs ~line 579-587) — they are NOT in GetCustomerList(). So the guest's spray loop iterates an empty/inert list. There is no SprayRequest MsgType and no guest->host forward for spraying (confirmed: grep for Spray/Deodorant/SmellyMeter across CoopCore.cs, Net/Msg.cs, Patches/GamePatches.cs returns nothing). NpcSync DOES mirror the Smelly flag host->guest (NpcSync.cs line 125/460/563), so the guest SEES the smelly FX but cannot act on it. Nothing blocks the guest from entering HoldSprayMode either, so the can depletes locally (DepleteContent, line 1626) for zero effect.

**Fix:** Add a guest->host forwarded spray op. Patch the client-side spray fire in InteractionPlayerController.RaycastHoldSprayState so the guest still plays local VFX and depletes the can for feedback, but instead of iterating its own empty GetCustomerList() it forwards a SprayRequest carrying the spray world-position, range (2.5f), and potency (1). Add the SprayRequest MsgType to Net/Msg.cs and a dispatch case in CoopCore that runs the DeodorantSprayCheck loop over the HOST's real GetCustomerList(), so the authoritative m_SmellyMeter decrements and the customer actually gets cleaned; the cleaned state then flows back to the guest via the existing NpcSync Smelly display flag. This mirrors the existing host-authoritative pattern already used for the AutoCleanser (which cleans host-side and mirrors the flag back).

## Deferred backlog (minor / cosmetic - NOT fixed yet)

Catalog divergence from differing content packs is an explicitly knownAccepted scenario for orders, so it is reachable here too.
- **[minor] Scanner-restock unlock (warehouse upgrade)** - The scanner-restock unlock purchase (ScannerRestockScreen.OnPress... line ~111: sets CPlayerData.m_IsScannerRestockUnlocked=true and fires ReduceCoin(m_UnlockCost)) is NOT patched by the mod. On the guest it charges the shared wallet (ReduceCoin forwarded, real host debit) and sets the guest's LOCAL flag true, but the unlock is never forwarded to the host. The scanner flag is only ever pushed HOST->client inside LicenseState (CoopCore.cs:2250 reads the HOST's flag; guest applies it OR-merged at 3036 CPlayerData.m_IsScannerRestockUnlocked |= scanner). Because the merge is one-directional (host's value into guest) and OR-only, the guest's self-unlock can never propagate to the host, and since guest saves are blocked the unlock evaporates on rejoin.
- **[minor] Deco-object purchase (buy into decoration inventory)** - A guest buying a deco object (ShopBuyDecoUIScreen.OnPressBuyShopDecoItem, category 3) fires CEventPlayer_ReduceCoin(price) - forwarded to the shared wallet - and AddDecoItemToInventory(+1) purely locally on the guest. m_DecorationInventoryList is a CPlayerData static that no sync module mirrors, so the host's inventory count never increments. The shared wallet paid but only the guest's mirrored inventory shows the item.
- **[minor] Customers (purchase interest from nearby pack-opening)** - When the GUEST opens a card pack near a host customer, the host's customers get no buying-interest boost. The guest's pack-open is broadcast only as MsgType.Activity, which CoopCore handles (CoopCore.cs:2681-2690) as a purely cosmetic avatar tag + ShowPackOpen animation - it never calls PlayerFinishOpenCardPack on the host, and the host has no knowledge of which of its customers the guest is standing near (the guest is a relayed avatar, not a position the customer AI reads).
- **[minor] Checkout / register serving** - The guest can still click a physical cashier counter and enter the game's native cash-counter mode (InteractableCashierCounter.OnMouseButtonUp is never blocked on the client). On the guest the counter's m_CurrentCustomer is null (customer sim is host-only and swept inactive), so there is nothing to scan or pay, but OnMouseButtonUp sets the guest's LOCAL m_IsMannedByPlayer=true and calls InteractionPlayerController.SetPlayerPos(m_LockPlayerPos.position) + OnEnterCashCounterMode, snapping the guest's camera to the empty register and locking movement until they press a move key / ESC (OnPressEsc).
